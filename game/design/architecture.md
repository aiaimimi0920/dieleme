# 云端技术架构 (Cloud Architecture)

为了实现极低的运维成本和纯静态部署，我们将采用 **"预计算 + 数据分片" (Pre-computation & Sharding)** 策略。

## 1. 核心挑战：大数据量 (The Data Scale Challenge)
*   **总量**: 约 200,000 条法拍记录。
*   **原始体积**: 约 100MB - 200MB。
*   **问题**: 浏览器无法一次性加载 200MB 数据，会导致页面卡死。

## 2. 解决方案：数据分片 (Data Sharding Strategy)
我们将数据按 **行政区划** 进行物理切割。用户只会下载通过“城市-区县”选择器定位到的那 **0.1%** 的数据。

### 2.1 文件结构 (File Structure)
```
/public/data/
  ├── meta.json             # [5KB]  元数据：包含所有城市列表、更新时间、全局统计 (用于首页)
  ├── cities/
  │   ├── 110100.json       # [20KB] 北京市索引：包含北京所有区县的统计数据
  │   ├── 310100.json       # [20KB] 上海市索引
  │   └── ...
  └── districts/
      ├── 110105.json       # [50KB] 北京市朝阳区：包含该区所有小区的详细数据
      ├── 310115.json       # [80KB] 上海市浦东新区
      └── ...
```

### 2.2 加载流程 (Loading Flow)
1.  **首页**: 加载 `meta.json`。用户看到“全国收录 200,123 套”，选择“上海”。
2.  **城市页**: 加载 `cities/310100.json` (上海)。用户看到浦东新区跌幅排名，点击“浦东”。
3.  **游戏页**: 加载 `districts/310115.json` (浦东)。
    *   此时浏览器只并下载了 **约 100KB** 的数据。
    *   **搜索**: 在这 100KB 数据中进行模糊搜索（如“世茂滨江”），响应速度 < 1ms。

## 3. 搜索优化 (Search Optimization)
如果用户想直接搜“世茂滨江”而不选城市怎么办？
*   **方案**: 我们生成一个 `search_index.json`，只包含 `{ "name": "世茂滨江", "city": "310100", "district": "310115" }`。
*   即便 20万条索引，gzip压缩后也只有 **约 2-3MB**。
*   **懒加载**: 只有当用户点击顶部搜索框时，才异步加载这个索引文件。

## 4. 构建流水线 (Build Pipeline)
我们编写一个 Python 脚本 `scripts/build_data.py`：
1.  读取 `datas/*.json` (原始抓取数据)。
2.  清洗、去重、计算跌幅/趋势。
3.  按照上述目录结构，将数据 Dump 为数百个小的静态 JSON 文件。
4.  最后生成 `web-app`。

## 5. 结论
**完全可行**。
对于 20万 级别的数据，分片后的静态托管是 **性能最好 (CDN边缘交付)** 且 **成本最低 (0元)** 的方案。
